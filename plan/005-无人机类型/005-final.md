# 无人机类型正式方案

## 1. 目的

本文件用于固化 `plan/005-无人机类型/005-draft.md` 与 `plan/005-无人机类型/005-answer.md` 讨论后的最终结论，作为对 `plan/001-总体设计/001-final.md` 中“节点类型映射”相关内容的增量修订。

本文件生效后，无人机类型不再按草案中的“历史轨迹长度 + 距离阈值 + 运动判断”处理，而采用更简单、可直接落地的归一化规则。

## 2. 修订范围

- 修订 `plan/001-总体设计/001-final.md` 中“节点类型固定为 8 类”的约束。
- 在现有 8 类节点类型之外，新增 1 个特殊类型：无人机。
- 本次只修订类型识别、高度归一化、图例/显隐/测试基线和文档同步要求。
- 本次不修改 `005-draft.md`，`005-draft.md` 仅保留为讨论草案。

## 3. 最终业务规则

### 3.1 无人机类型值

- 无人机类型值固定为 `30`。
- 本阶段不做运行时可配置，不增加环境变量入口。
- 后续代码中应使用统一常量，避免在多个文件中散落硬编码 `30`。

### 3.2 无人机识别方式

- 无人机类型不从后端 `node_type` 字段直接获得。
- 无人机识别基于节点的 `node_manage_ip_addr`。
- 当节点 IP 命中无人机名单时，前端应忽略原始 `node_type`，直接将节点类型覆盖为无人机类型 `30`。
- 当节点 IP 未命中无人机名单时，沿用后端原始 `node_type`。

### 3.3 IP 名单承载方式

- 本阶段采用代码内硬编码名单，不走环境变量配置。
- 这样做的原因是当前项目配置体系仅支持字符串、数字、布尔等简单值，尚未提供 IP 列表或 IP 区间的稳定解析能力。
- 正式实现时可在代码中维护“无人机 IP 白名单集合”，直接按完整 IP 字符串匹配。

当前确认的无人机 IP 来源范围如下：

- `172.16.6.11` 至 `172.16.6.18`
- `172.16.7.20` 至 `172.16.7.30`
- `172.16.8.32` 至 `172.16.8.42`
- `172.16.9.44` 至 `172.16.9.73`

落地时应将上述范围展开为完整 IP 白名单，再进行精确匹配。

### 3.4 高度归一化规则

- 无人机节点在归一化阶段统一处理高度。
- 当节点已被识别为无人机，且解析后的高度值小于 `1` 时，直接将高度改为 `100`。
- 当节点已被识别为无人机，且解析后的高度值大于等于 `1` 时，保留原始高度。
- 非无人机节点不应用该规则。

### 3.5 删除的旧规则

以下规则全部废弃，不再进入正式实现：

- 不再判断历史轨迹列表长度。
- 不再判断当前位置与上一轨迹点的距离。
- 不再根据“是否运动”决定是否补高度。
- 不再在“加入历史轨迹之前”单独补高度。

## 4. 归一化阶段落点

- 无人机类型覆盖必须在拓扑响应归一化阶段完成。
- 无人机高度补偿也必须在同一个归一化阶段完成。
- 归一化完成后，`SituationNode.type` 与 `SituationNode.height` 应已经是最终值。

这样处理的原因是：

- 当前地图上的节点位置显示依赖归一化后的节点数据。
- 当前历史轨迹也是基于归一化后的节点数据追加。
- 当前图例、显隐、标牌颜色和节点图标样式也依赖归一化后的节点类型。

因此，只有在归一化阶段一次性完成“类型覆盖 + 高度修正”，才能保证当前点位、历史轨迹、图例和交互状态全部一致，避免前后层数据不一致。

## 5. 图标与样式要求

- 无人机图标使用 `public/models/node-icons/svg/无人机.svg`。
- `nodeTypeStyle.ts` 需要新增无人机类型样式配置。
- 图例面板需要展示无人机类型。
- 类型显隐逻辑需要支持无人机类型。
- 节点标牌与连接线配色逻辑需要支持无人机类型。

本次新增后，系统中的“已知类型集合”将从 8 类扩展为 9 类，不再维持“固定 8 类”的旧约束。

## 6. 对现有设计与测试的影响

### 6.1 对总体设计文档的影响

- `plan/001-总体设计/001-final.md` 中“节点类型固定为 8 类”的表述需要同步修订。
- 与“8 类图例”“8 类节点样式”“8 类验收项”相关的内容都需要同步更新为包含无人机类型的新口径。

### 6.2 对实现代码的影响

正式实现至少应覆盖以下位置：

- 拓扑归一化逻辑：增加无人机识别和高度归一化。
- 类型样式配置：新增无人机类型样式项。
- 图例渲染：从 8 项调整为 9 项。
- 类型显隐：新增无人机类型键值支持。
- 节点信息标牌：显示无人机类型名与对应样式。

### 6.3 对测试基线的影响

以下旧断言需要同步修订：

- “已知节点类型为 8 类”
- “图例渲染 8 项”
- “节点类型列表固定为 1-8”
- “节点样式配置长度为 8”

同时应新增与无人机规则相关的测试：

- IP 命中名单时，节点类型被覆盖为 `30`
- IP 未命中名单时，保留原始 `node_type`
- 无人机节点高度小于 `1` 时被改为 `100`
- 无人机节点高度大于等于 `1` 时保持原值
- 图例、显隐和节点样式能够识别无人机类型

## 7. 实施原则

- 优先最小改动原则，不引入额外的复杂配置系统。
- 优先单一落点原则，把规则集中在归一化阶段处理。
- 优先一致性原则，确保节点当前位置、历史轨迹、图例和显隐逻辑使用同一份归一化结果。
- 优先文档同步原则，代码完成后必须同步修订总体设计、详细开发步骤和相关验收文档。

## 8. 最终结论

本次无人机类型方案按以下规则执行：

- 无人机类型值固定为 `30`
- 无人机识别依据为 `node_manage_ip_addr`
- 无人机 IP 名单本阶段在代码中硬编码
- 命中无人机名单后覆盖原始 `node_type`
- 无人机高度小于 `1` 时在归一化阶段直接改为 `100`
- 不再保留历史轨迹长度、距离阈值和运动判断逻辑
- 图例、显隐、测试基线和文档必须同步扩展到 9 类口径
